GO的rpc包中Server的实现

前面的一篇博客分析了一下关于GO的rpc中关于client的实现，大致理了一下client发送请求并且如何收到server的回复的一个过程。当时准备立即去分析server.go的代码的，但是因为GO中的泛型的一些知识比较多，于是简单总结了一下泛型的知识点。

有了泛型的知识，我们在分析server的时候就可以更加集中的去看与rpc相关的内容，这里我还还是坚持我们在看源码的时候不要想着将每一行的代码都看的明白，因为源码中会因为程序的健壮性、稳定性、通用性等原因加入大量的与业务不是很相关的代码。不是说这部分不重要，而是说我们要有一个全局的视角，如果我是想了解rpc的过程，适当忽略错误代码或者是特殊情况处理的代码是可以的，如果我要去学习相关的错误处理、提升自己程序的稳定性那我们就去重点关注相关的错误处理和特殊情况处理的逻辑代码，一定要找到阅读源码的重点。

读源码就类似于我们读文言文、读英文文章一样，不要纠结于某一个字某一个单词，先整体贯通，最后看需要去落实相关的一字一句，否则你读源码会很累，而且很难读下去。

总结一下这一段代码的流程：

  1. 创建装在receiver方法的map
  2. 遍历方法集
  3. 如果方法符合rpc对方法的规范就加入，否则continue进入下一个的处理
  4. 返回这个方法集（service会将会接收这个map）

这个方法就再次应征了一个道理，这些对方法的细节检查处理与rpc过程关系不大，如果你了解rpc方法规范这些都是小问题。但是对于我们反射的学习，这里又是非常值得我们学习的代码，所以我们一定要处理好源码阅读的取舍，在后面的博客中我也会坚持这个原则。

总结

这就是一个最简单的server处理client的rpc请求的过程，重点有以下几个：

  1. server如何注册相关的服务（service）：利用反射获取receiver的Type，用一个map[serviceName]service来管理各种服务
  2. 如何添加service地那些方法：利用反射扫描service的Type中的方法集，找出符合rpc规范的方法
  3. service如何管理这些方法：利用map[methodName]methodType这个有方法名到方法的映射
  4. 如何正确确定client需要调用的方法：利用service.method这个路径唯一确定一个方法

好了，关于rpc调用的过程client和server都算是有了，但是这也只是一个很简陋的调用模型，如果我们需要构建大规模的rpc调用程序，仍然需要考虑到更多的东西。例如：

  • 我们的介质是不是只有tcp连接一种，http是否可以？当然可以，并且我们还有更好的gRpc
  • 这里是gob编码格式的codec，如果想实现json或者是xml可不可以？可以，只要你能够实现Codec需要的接口
  • 如何制定更规范的接口规范，我们的client与server都是通过我们的定义的service方法来交互，这个接口的规范制定要严谨

所以博主说过，我们在学习rpc的原理的时候要蔑视它，但是深入的时候需要怀着敬畏之心去学习，我们看到的高楼大似乎都很简单，就跟我们垒砖块差不多，但是这里面的复杂我们也要知道，垒得好就能屹立不倒，不然就是害人害己。对于rpc应用的构建，这个道理是相通的！
