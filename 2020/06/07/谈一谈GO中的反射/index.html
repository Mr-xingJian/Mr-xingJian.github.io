谈一谈GO中的反射
Posted on 2020-06-07   Updated on 2020-06-13 GO reflect

Reflect

无奈的反射

博主之前研究过Java的反射，有过一点反射概念的基础，所以在面对GO中的基本的反射操作是能够看得懂的。所以在前面两篇rpc的文章，都进展的比较顺利。但是今天看到rpc包中的server的时候，因为server要处理各种各样的rpc请求，考虑到通用性，其中涉及到了大量的反射，再强行看下去实在是有点头疼，所以找个时间来补一补GO中反射的知识，也算是无奈的反射，不得不反射了。

Type和Value

在静态语言中，Type和Value就像是天经地义的事情，毕竟你总不能搞出一个没有类型的变量，在GO中即使:=自动推导时我们没有指明Type，但是我们依然知道自动推导会帮我们指明类型，所以这个变量并非无类型。

1.Type

  • reflect.TypeOf(interface{})可以接收任何interface{}参数，并将其类型以reflect.Type的形式返回

2.Value

  • reflect.ValueOf(interface{})则可以将任何interface{}参数的动态值以reflect.Value形式返回
  • reflect.Value的Kind方法可以将无限多的类型分成有限的几种返回，分别是Bool、String、Number（Int8……）、Array、Struct、Chan、Func、Ptr、Slice、Map和Interface，最后还有一种特殊的Invalid（reflect.Value的零值就是Invalid）

刚开始接触反射的时候我也很不适应，既然静态语言类型都是确定的，我们为什么还需要探索变量的类型和值呢？因为你不是所有时候都一定知道类型和值，例如你想实现一个函数可以处理多个类型的参数，那么一般我们会用interface{}作为载体，在具体时间的时候我们肯定要经过Type的判断，当然这种时候我们更多的是用类型断言。

挑几个重要的方法说一说

Type（1-9是通用的，之后的是某种Type特有的）

  1. Method(int) Method，返回具有方法集的第i个方法的描述，注意方法的参数的第一个是receiver，如果是interface{}则没有receiver，第一个参数为nil

  2. MethodByName(string) (Method, bool)，第二个参数表示有没有这个方法

  3. NumMethod() int，方法集的个数

  4. PkgPath() string，这里需要说明一下

    • 这个函数会返回一个Type的import package path，例如encoding/json，可以唯一确定一个包
    • 有些类型例如string，int这种内建类型，返回空字符串
    • 未定义的类型，例如*T，struct{}，[]int同样返回空字符串
    • 如果一个A只是一个类型的别名，例如type A = int（type Arr = []int），同样返回空字符串

  5. Kind() Kind，返回kind of type，Kind是一个所有类型的枚举变量

  6. Implements(u Type) bool，很简单，一个Type有没有实现某一个接口（想一想这是如何做到的！！！）

  7. AssignableTo(u Type) bool，一个Type是否可以赋值给另一个

  8. ConvertibleTo(u Type) bool，一个Type是否可以转型成另一个

  9. Comparable() bool，一个Type是不是可比较的

  10. Bits()，返回一个Type占多少个bit，适用于Int\, Uint*, Float*, Complex*这些类型

  11. ChanDir() ChanDir，返回一个channel的方向，当然只能适用于Chan

  12. IsVariadic() bool，返回一个函数最后一个参数是不是可变长参数，适用于Func

  13. Elem() Type，放回一个Type的元素类型，适用于Array, Chan, Map, Ptr, or Slice

  14. Field(i int) StructField，返回结构体的第i个Field，适用于Struct

  15. FieldByIndex(index []int) StructField

  16. FieldByName(name string) (StructField, bool)，第二个参数代表有没有这个Field

  17. FieldByNameFunc(match func(string) bool) (StructField, bool)，使用match方法来匹配

  18. NumField() int，返回struct的Field个数，适用于Struct

  19. In(i int) Type，返回函数第i个参数的Type，适用于Func

  20. Out(i int) Type

  21. NumIn() int，返回函数参数的个数，注意receiver的存在，适用于Func

  22. NumOut() int

  23. Key() Type，返回map的key的Type，适用于Map

  24. Len() int， 返回Type的长度，适用于Array

博主认为，反射最重要的有两点，一是理解反射的概念，为什么我们需要Type和Value，二是要多实践，这个实践有包括两个点，看别人的反射代码和自己去写反射代码。这篇博客我会从这两个方面呈现我个人对反射的理解。

关于通用性

无论什么时候你一定用过fmt.Printf这个东西吧，无论是想要打印什么东西还是想要输出一点调试信息，我们都离不开它。对于printf最早的记忆应该是C语言，我们的第一个hello world可能就是printf输出的。在C语言中，f（format）的要求是非常严格的，比如说%d对应了int、%c对应了char、%f对应了float，printf会按照我们的格式化的占位符将信息打印出来。但是在C语言中你应该不能直接打印一个结构体吧？甚至是一个数组也不可以，GO就不一样了，除了具体格式化的类型，slice、map、struct没有GO不可以打印的，这就是你平时经常用的%v占位符（它几乎能代表任何一种Type）。这么神奇的东西就是由泛型在支持着它，所以我们的fmt.Printf的通用性也就大大增加，那么通用性就从fmt.Ptintf中的%v占位符开始！

因为后面涉及到东西，例如内部一些工具对象的获取什么的，这不是我们几天的重点，让我们加一加速，直接到达%v，重点关注reflect！！！

在这个过程中的doPrintf函数中我们终于见到了，判断如果我们的格式字符c是v的话就执行p.printArg(a[argNum], rune(c))，话不多说，直接进入这个函数，先搞清楚参数的含义，分别是承载真实变量的interface{}和格式化字符，这里面会处理一些列的特殊情况以及不用反射就能完成的类型，最后我们会去到p.printValue(f, verb, 0)，来看一看：

fmt.Printf最终是将我们的符合类型的在反射reflect.Value的基础上不断地判断value的类型，知道找到基本类型，给出正确的处理方法。fmt包的作者在后面为我们做出了如此多的努力，才让我们可以毫不费力的使用%v这个东西，这段代码不仅是很好的展示了反射的用法，更是很好的展现了递归的魅力，将确定基本类型作为递归出口值得我们学习！

总结一下

将data抽象出Type和Value意味着我们几乎可以对GO中一切进行抽象，我们有了在最高层设计我们的程序的视角！

另外这三个反面也并没有一个非常明显的区别，大多数的时候都是杂糅在了一起，这就是反射的魅力，但是说实话要真的用好反射还是需要一定的时间的，博主现在也用不好。一是我们还没有达到可以将data抽象成Type和Value的这种意识高度，即使有了试用的场景我们也很难意识到，二是如果我们都是做一些简单的小项目确实遇不到反射的场景，强行反射并不优雅！
