初探go test
Posted on 2020-06-30   Updated on 2020-07-01 GO Test

go test

这段时间的工作一直主要是GO，很多业务代码写完之后在考虑加上单元测试，就来研究一下go test这个工具

go test是一个按照一定的约定和组织来测试代码的程序，包中的_test.go结尾的源文件不会被构建成报的一部分，它们属于go test的测试对象

go test文件中主要包含三种类型的函数：

  • 测试函数：Test开头，测试程序的逻辑行为
  • 基准测试函数(benchmark)：Benchmark开头，测试函数的性能，原理是go test会多次执行基准函数计算平均执行时间
  • 示例函数：Example开头，提供一个由编译器保证正确性的示例文档

go test执行流程：

  • 遍历所有*_test.go文件中的符合测试函数名称规范的函数
  • 生成一个临时的main包用于调用测试函数
  • 构建并运行，报告测试结果
  • 清理测试过程生成的临时文件

example：编写测试函数测试斐波那契函数

测试命令

  • go test：会给出相应的测试结果，已经这个测试的时间
  • go test -v：当n调至40的时候明显看出，递归与迭代之间的速度差距
  • go test -run=[regexp]：run参数可以指定一个正则表达式，只测试匹配的函数

测试表格

单个测试用例可能并不能完全说明我们测试的正确性，有时候需要提供多个测试用例，这个时候可以使用测试表格加循环迭代来简化

白盒测试

黑盒测试注重的是结果，我们一般会调用相关的API，然后看输出是否符合预期，就像选择题，我只看你结果对不对。白盒测试又成为clear box，一般会进入包的内部，需要观察函数的执行流程对不对，对于白盒测试，可就不是简单的条条大路通罗马了，我们希望从特定的路线达到正确的目的。

从上面可以看出，第一个测试案例是一个黑盒测试，因为我们只对包导出的函数进行了测试，第二个就不一样了，我们对包内的函数进行了相关的测试，另外我们拿到了同一个包内的变量进行了相关的使用，这就是典型的白盒测试。

另外需要注意的就是默认的go test命令并不会执行什么benchmark函数，我们需要通过go test -bench=[regexp]来匹配需要测试的benchmark函数，go test -bench=.可以测试所有的benchmark函数。

虽然有了这样的性能测试工具，但是我们千万不要再开发过程中去过于的纠结性能，因为benchmark测试也只是一种参考，为了很小很小的性能去复杂代码其实不值得。不过话说回来，如果是向上面这样区别巨大，那么还是值得我们去优化的。

By the way

大多数时候其实编写真正的单元测试代码是没有这么容易的，因为往往会有很多的东西耦合起来，会涉及到各种各样的依赖注入问题，这样会导致你写一个测试可能需要解决各种各样的依赖问题，比较棘手。

这也在侧面体现了你的代码能力，好的代码必须要能够尽可能简单的测试，在平时写代码的过程中我们也要注意这一点，另外一方面编写单元测试也是需要很多经验和技巧的，要在实践中慢慢的积累。
